# Як повинен працювати ffmpeg у проєкті Emotional TTS

## 1. Роль ffmpeg
`ffmpeg` у цьому проєкті не займається семантикою тексту і не визначає емоції. Його задача — **технічна обробка аудіо після синтезу**.

## 2. Основні функції ffmpeg у системі
ffmpeg потрібен для:
- склеювання сегментів;
- конвертації форматів;
- нормалізації гучності;
- обрізання або керування тишею;
- уніфікації параметрів аудіо.

## 3. Місце ffmpeg у pipeline

```text
Piper segment WAV files (через TTS adapter)
  -> ffmpeg concat / processing (усередині tts-adapter)
  -> final wav
  -> optional mp3 conversion
  -> output for UI/download
```

## 4. Що саме має робити ffmpeg у MVP

### 4.1 Склеювання сегментів
Після сегментного синтезу потрібно зібрати один файл з:
- WAV сегментів;
- додаткових пауз між ними, якщо вони потрібні;
- правильного порядку сегментів.

### 4.2 Нормалізація гучності
Потрібно зробити так, щоб результат не звучав занадто тихо або занадто нерівно між сегментами.

### 4.3 Конвертація формату
Базовий внутрішній формат варто тримати як `wav`, а фінальний клієнтський формат — `mp3` або `wav` за вибором.

### 4.4 Контроль тиші
За потреби можна:
- додавати короткі паузи;
- прибирати зайві хвости;
- уникати неприродних розривів між сегментами.

## 5. Чому ffmpeg не можна змішувати з parser logic
Тому що ffmpeg працює з **аудіосигналом**, а parser працює з **текстовими сигналами**. Це різні рівні системи.

Правильний поділ такий:
- parser вирішує, **де** потрібна пауза;
- ffmpeg реалізує, **як технічно** ця пауза вставляється або як склеюється результат.

## 6. Архітектурна роль ffmpeg
ffmpeg загорнуто в **audio utility layer** всередині сервісу `tts-adapter` (Python). Саме цей сервіс відповідає за:

- `concatSegments()`
- `normalizeLoudness()`
- `convertFormat()`
- `trimSilence()`

## 7. Структура коду у поточному каркасі
```text
src/
  services/
    tts-adapter/
      postprocessing/
        ffmpeg_runner.py
        ffmpeg_command_builder.py
        concat.py
        normalize.py
        convert.py
      domain/
        audio_job.py
        audio_result.py
```

## 8. Правила використання

### Правило 1. Внутрішній базовий формат — WAV
WAV варто використовувати як технічний проміжний формат, бо він простий і передбачуваний для злиття та подальшої обробки.

### Правило 2. Конвертація в MP3 — лише наприкінці
Не потрібно конвертувати кожен сегмент окремо в MP3 до склеювання.

### Правило 3. Усі ffmpeg-команди мають бути централізовані
Не можна розкидати shell-команди по всьому коду gateway або front-end.

### Правило 4. Обробка файлів має бути ізольована на рівні requestId
Усі тимчасові та фінальні файли запиту мають лежати в окремому каталозі.

## 9. Мінімальні операції для MVP

### concatSegments
Вхід:
- список WAV-сегментів у порядку синтезу.

Вихід:
- один `final.wav`.

### normalizeLoudness
Вхід:
- `final.wav`.

Вихід:
- `final_normalized.wav`.

### convertFormat
Вхід:
- `final_normalized.wav`.

Вихід:
- `final.mp3`.

## 10. Де викликати ffmpeg
Правильне місце виклику в поточній архітектурі:
- не front-end;
- не Python parser;
- не безпосередньо в gateway;
- а **post-processing шар сервісу tts-adapter**, який запускається після етапу синтезу.

## 11. Логіка обробки пауз
Паузи повинні визначатися на рівні metadata, наприклад:
```json
{
  "pauseAfterMs": 180
}
```

Після цього audio utility layer реалізує це через:
- вставку тиші;
- підготовлений concat-ланцюг;
- або іншу технічну схему в рамках одного сервісу.

## 12. Помилки, які треба обробляти
- відсутній ffmpeg у середовищі;
- пошкоджений або порожній WAV-сегмент;
- невірний формат вхідного файлу;
- збій під час concat;
- збій під час конвертації.

## 13. Які артефакти зберігати
Для дебагу корисно зберігати:
- список сегментів;
- файл-збірку для concat;
- `final.wav`;
- `final.mp3`;
- лог або JSON зі статусом post-processing.

## 14. Що не повинен робити ffmpeg-рівень
- не повинен самостійно вибирати emotion label;
- не повинен переосмислювати metadata parser-а;
- не повинен вирішувати бізнес-логіку режимів `neutral / expressive`;
- не повинен бути єдиним джерелом інформації про стан задачі.

## 15. Критерії готовності ffmpeg pipeline
ffmpeg pipeline вважається готовим, якщо:
- склеює сегменти у правильному порядку;
- вирівнює фінальне аудіо до стабільного результату;
- конвертує у вибраний формат;
- працює відтворювано в Docker-середовищі.
