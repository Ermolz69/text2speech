## Обзор
Этот документ определяет roadmap реализации **эмоционально-осознанной системы text-to-speech**, которая принимает текст с эмодзи и пунктуационными подсказками, интерпретирует эмоциональное намерение и локально генерирует выразительную речь.

Roadmap рассчитан на **команду из 7 человек** и опирается на следующее стратегическое решение:

- **Основной локальный TTS-движок:** Piper
- **Главная цель MVP:** локальный, бесплатный, Docker-friendly деплой
- **Главный продуктовый вызов:** не построение новой TTS-модели с нуля, а создание **слоя управления** поверх TTS, который превращает текстовые эмоциональные сигналы в выразительное поведение синтеза
- **Цель бенчмарка:** сравнить локальный результат с более качественным облачным эталоном, например ElevenLabs

См. [Ключевые решения](#ключевые-решения) и [Технологический стек](#технологический-стек).

## Постановка задачи
На вход подаётся обычный текст, который может содержать эмодзи, усиленную пунктуацию, повторяющиеся символы и контекстные маркеры сентимента. Система должна:

1. обнаруживать эмоциональные сигналы в тексте,
2. преобразовывать их в структурированные эмоциональные метаданные,
3. переводить метаданные в инструкции для синтеза,
4. генерировать аудио через локальный TTS-движок,
5. оценивать, стала ли полученная речь более выразительной по сравнению с нейтральным синтезом.

Это в первую очередь задача **text-to-speech**, а не speech-to-text. Speech-to-text может быть добавлен позже только для бенчмаркинга, голосового ввода или оценки на основе транскрипции. См. [Стратегия бенчмаркинга](#стратегия-бенчмаркинга).

## Границы продукта
### Входит в scope
- Текстовый ввод с эмодзи и пунктуацией
- Парсинг эмоций и rule-based интерпретация
- Эмоциональные метаданные на уровне сегментов
- Локальный синтез через Piper
- Dockerized local development flow
- Базовый UI или API для тестирования
- Объективная и субъективная оценка
- Бенчмаркинг относительно более сильной внешней reference-системы

### Не входит в MVP
- Обучение новой эмоциональной TTS-модели с нуля
- Полный end-to-end STT + TTS voice assistant workflow
- Продакшн-уровневая распределённая инфраструктура
- Voice cloning как основной артефакт
- Real-time conversational streaming как жёсткое требование

## Цели и нецели
### Цели
- Быстро получить рабочий локальный MVP
- Сохранить возможность бесплатного локального запуска на старте
- Отделить интерпретацию эмоций от синтезирующего backend
- Сделать пайплайн измеримым и пригодным для бенчмаркинга
- Построить кодовую базу, в которой позже можно заменить Piper на другой движок

### Нецели
- Превзойти state-of-the-art облачный эмоциональный TTS в первой итерации
- Идеально решить все случаи сарказма и иронии в рамках MVP
- Полностью зависеть от платного вендора с первого дня

## Ключевые решения
1. **Использовать Piper как основной слой синтеза.** Piper — это быстрый локальный neural TTS engine, а актуально поддерживаемый проект находится у OHF-Voice. Текущий README также описывает CLI, web server, Python API и training documentation, что делает Piper практичным локальным baseline.
2. **Не делать fine-tuning и не переобучать TTS-модель на первой фазе.** Сначала нужно построить слой управления вокруг модели.
3. **Представлять эмоцию явно в виде метаданных.** Парсер должен возвращать структурированную информацию, а не только факт обнаружения эмодзи.
4. **Использовать Docker для воспроизводимости.** Каждый участник команды должен иметь возможность поднять одинаковый локальный стек.
5. **Использовать librosa для оценки, а не для синтеза.** librosa полезна для извлечения признаков вроде MFCC, RMS, spectral centroid, spectral bandwidth и spectral contrast.
6. **Использовать ElevenLabs только как benchmark и reference по качеству.** Он полезен для сравнения, но не является лучшим выбором для бесплатного local-first MVP.
7. **Сохранять provider-agnostic архитектуру.** Слой TTS должен быть достаточно абстрактным, чтобы позже можно было подключить ElevenLabs, Azure, Coqui или другого провайдера.

## Видение системы
Система должна работать как пайплайн, который превращает текстовые сигналы в выразительное поведение синтеза.

```text
User text
  -> normalizer
  -> segmenter
  -> emoji and context parser
  -> emotion mapping
  -> prosody planner
  -> TTS adapter
  -> audio post processing
  -> output audio + metrics
```

Самый важный архитектурный принцип — разделение ответственности:
- понимание текста должно быть независимым от движка синтеза,
- движок синтеза должен быть заменяемым,
- оценка должна быть воспроизводимой.

## Целевая архитектура
Архитектура должна включать следующие сервисы или модули:

1. **Клиентский слой**
   - простой UI, CLI или API consumer,
   - отправляет текстовый ввод и получает аудио.

2. **Сервис парсинга и управления**
   - нормализует текст,
   - извлекает сигналы,
   - генерирует эмоциональные метаданные.

3. **TTS-сервис**
   - принимает текстовые сегменты и synthesis hints,
   - синтезирует речь через Piper.

4. **Слой аудио-утилит**
   - преобразует WAV в MP3 при необходимости,
   - склеивает сегменты,
   - нормализует громкость.

5. **Слой оценки**
   - рассчитывает объективные метрики,
   - хранит benchmark outputs,
   - поддерживает listening tests.

См. [Ключевые модули](#ключевые-модули) и [Как система должна работать end-to-end](#как-система-должна-работать-end-to-end).

## Ключевые модули

### Входной слой
Задачи:
- принимать обычный текст,
- принимать текст с эмодзи,
- принимать текст с пунктуационными паттернами вроде `!!!`, `???`, `...`,
- при необходимости принимать batch test files.

Начальные варианты интерфейса:
- CLI для разработчиков,
- минимальная web form для демо,
- REST endpoint для автоматизации.

### Нормализация и сегментация
Задачи:
- нормализовать пробелы,
- сохранять эмодзи,
- разбивать текст на предложения или смысловые куски,
- обнаруживать повторяющуюся пунктуацию,
- помечать uppercase emphasis.

Выход этого модуля должен быть нормализованным внутренним представлением, а не финальными метками эмоций.

### Парсер эмодзи и контекста
Задачи:
- обнаруживать эмодзи,
- обнаруживать пунктуационные усилители,
- обнаруживать слова-модификаторы вроде “очень”, “так”, “сильно”,
- определять расположение сигналов относительно предложения,
- понимать, является ли сигнал глобальным или локальным.

Этот модуль должен отвечать на вопросы:
- какие эмоциональные сигналы существуют,
- к чему они применяются,
- насколько сильными они выглядят.

### Слой отображения эмоций
Задачи:
- переводить обнаруженные сигналы в небольшое управляемое пространство меток,
- назначать уровни интенсивности,
- комбинировать эмодзи, пунктуацию и лексические сигналы.

Рекомендуемый начальный набор меток:
- neutral
- joy
- playful
- sadness
- anger
- surprise

Рекомендуемая шкала интенсивности:
- 0 neutral
- 1 weak
- 2 medium
- 3 strong

Именно здесь должна находиться первая версия rule logic.

### Слой планирования просодии
Задачи:
- переводить emotional metadata в synthesis hints,
- определять границы сегментов,
- выбирать длину пауз,
- настраивать стратегию выделения,
- готовить chunks, готовые к синтезу.

Важное замечание: в MVP этот слой в основном управляет **тем, как подготавливается текст** к синтезу. Он не требует настоящего внутреннего emotional control внутри Piper.

### Слой TTS-провайдера
Задачи:
- предоставлять единый интерфейс синтеза,
- оборачивать Piper как основного провайдера,
- позволять добавлять других провайдеров позже.

Рекомендуемый интерфейс:

```python
synthesize(text: str, voice_id: str, output_path: str, metadata: dict | None = None) -> str
```

Реализации провайдеров:
- `PiperProvider`
- позже `ElevenLabsProvider` только для benchmark
- опционально `AzureProvider` или `CoquiProvider` позже

### Постобработка аудио
Задачи:
- объединять аудиофайлы сегментов,
- нормализовать громкость,
- при необходимости подрезать края тишины,
- конвертировать выходной формат для воспроизведения.

Рекомендуемые инструменты:
- `ffmpeg`
- `pydub` или другой небольшой utility layer

### Оценка и аналитика
Задачи:
- вычислять аудио-признаки через librosa,
- сравнивать neutral и expressive outputs,
- хранить benchmark metadata,
- поддерживать listening tests.

Рекомендуемые метрики:
- длительность
- количество и длина пауз
- RMS energy
- summary statistics по MFCC
- spectral centroid
- spectral bandwidth
- spectral contrast

См. [Стратегия бенчмаркинга](#стратегия-бенчмаркинга).

## Технологический стек
### Целевой стек
- **Frontend:** React + TypeScript
- **Gateway backend:** TypeScript + Fastify
- **Сервис текстового анализа:** Python
- **TTS-сервис:** Piper
- **Хранилище / логи:** локально, позже при необходимости object storage

### Пояснения по выбору стека
- **Frontend** отвечает только за пользовательский интерфейс и ходит в gateway по API.
- **Gateway backend** агрегирует запросы, координирует вызовы сервиса текстового анализа и TTS-сервиса и предоставляет единую публичную точку входа.
- **Сервис текстового анализа** реализует нормализацию, парсинг эмодзи и контекста, отображение эмоций и логику планирования просодии.
- **TTS-сервис (Piper)** рассматривается как заменяемый синтезатор за простым адаптерным интерфейсом.
- **Хранилище/логи** на старте базируются на локальной файловой системе; при необходимости позже можно добавить S3-совместимое object storage для долговременных артефактов и метрик.

## Данные и тестовые входы
Нужен структурированный тестовый корпус с самого начала.

### Группы входных данных
1. **Нейтральные предложения**
   - без эмодзи,
   - с минимальным усилением пунктуации.
2. **Простые эмоциональные кейсы**
   - один эмодзи, ясное значение.
3. **Усиленные кейсы**
   - повтор эмодзи, повтор пунктуации, uppercase.
4. **Неоднозначные кейсы**
   - ирония, шаблоны сарказма, смешанные сигналы.
5. **Многосегментные кейсы**
   - длинный текст, где эмоция меняется по сегментам.

### Пример формата корпуса
Каждый тестовый элемент должен включать:
- input text,
- expected label,
- expected intensity,
- notes по неоднозначности,
- benchmark outputs.

## Как система должна работать end-to-end
1. Пользователь вводит текст.
2. Вход проходит через нормализацию и сегментацию.
3. Парсер обнаруживает эмодзи, пунктуацию и лексические усилители.
4. Слой отображения эмоций назначает метки и интенсивности.
5. Планировщик просодии превращает метаданные в synthesis-ready сегменты.
6. Piper синтезирует каждый сегмент.
7. Аудио-утилиты склеивают и нормализуют итоговый output.
8. Утилиты оценки при необходимости вычисляют метрики и сохраняют отчёты.

### Пример потока
Вход:

```text
I am so happy to see you 😊😊!
```

Промежуточные метаданные:

```json
{
  "segments": [
    {
      "text": "I am so happy to see you",
      "emotion": "joy",
      "intensity": 2,
      "emoji": ["😊", "😊"],
      "punctuation": ["!"],
      "pause_after_ms": 120
    }
  ]
}
```

Выход:
- один синтезированный аудиофайл,
- optional metrics file,
- optional comparison against neutral synthesis.

## Стратегия бенчмаркинга

### Бенчмарк A. Piper neutral vs Piper emotion-aware pipeline
Цель:
- доказать, что слой управления действительно меняет поведение output.

Метод:
- синтезировать каждый input дважды,
- один раз как plain synthesis,
- один раз с parsed emotional metadata,
- сравнить аудио-признаки и listening impressions.

Сигнал успеха:
- эмоциональный и нейтральный outputs слышимо различаются в нужном направлении.

### Бенчмарк B. Piper vs ElevenLabs
Цель:
- понять разрыв между бесплатным локальным MVP и более качественным облачным reference.

Метод:
- использовать одни и те же предложения и intended emotional labels,
- генерировать аудио через Piper и ElevenLabs,
- сравнивать naturalness, perceived emotion и latency.

Что фиксировать:
- latency синтеза,
- общую длительность аудио,
- субъективную оценку,
- заметки по воспроизводимости,
- заметки по стоимости и внешним зависимостям.

### Бенчмарк C. Синтетическое аудио vs человеческая эталонная запись
Цель:
- понять, насколько система приближается к естественному выразительному чтению.

Метод:
- записать человека, читающего те же prompts с заданными эмоциями,
- сравнить human audio, output Piper и benchmark output,
- по возможности провести blind listening tests.

Рекомендуемые оси оценки:
- naturalness,
- emotional clarity,
- intelligibility,
- listening comfort.

## Критерии приёмки
MVP считается принятым, если выполняются все следующие условия:

1. Система принимает текст с эмодзи.
2. Парсер выдаёт структурированные эмоциональные метаданные.
3. Метаданные воспроизводимо влияют на сгенерированное аудио.
4. Piper запускается локально в Docker для всей команды.
5. Как минимум один голос интегрирован и работает end-to-end.
6. Существует и версионируется benchmark corpus.
7. Реализовано базовое сравнение с нейтральным синтезом.
8. План benchmark против ElevenLabs задокументирован и хотя бы частично выполнен.
9. В репозитории есть инструкции по setup и demo path.

## Риски и меры снижения
### Риск 1. Неоднозначное значение эмодзи
Меры снижения:
- начать с небольшого набора меток,
- логировать неоднозначные кейсы,
- использовать явный fallback в neutral.

### Риск 2. Ограниченная выразительность Piper
Меры снижения:
- управлять выразительностью через сегментацию и synthesis hints,
- использовать benchmark-сравнения для выявления разрывов,
- заранее держать готовой provider abstraction.

### Риск 3. Ограничения языка или голосов
Меры снижения:
- рано зафиксировать известные поддерживаемые голоса для MVP,
- версионировать voice assets,
- рано провести валидацию качества голосов.

### Риск 4. Проблемы с локальной воспроизводимостью
Меры снижения:
- использовать Docker,
- фиксировать версии,
- рано документировать runtime setup.

### Риск 5. Переусложнение в ранних спринтах
Меры снижения:
- избегать retraining и сложных research features в MVP,
- сначала сосредоточиться на control layer.
