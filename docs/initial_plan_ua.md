# Поточний стан реалізації Emotional TTS MVP

## 1. Мета MVP, яку вже таргетує каркас
Поточний каркас репозиторію налаштовано під локальний MVP, який:
- приймає текст з емодзі та підсиленою пунктуацією;
- визначає емоційні сигнали;
- перетворює їх на структуровані metadata;
- синтезує аудіо через Piper;
- виконує базову post-processing обробку через ffmpeg;
- повертає користувачу готовий аудіофайл і службові метадані.

## 2. Що вже зафіксовано в репозиторії
На рівні roadmap та стартового коду вже визначено й частково реалізовано:
- основний локальний TTS-рушій — **Piper**;
- стек: **React + TypeScript**, **TypeScript + Fastify**, **Python**, **ffmpeg**, **librosa**;
- центральний pipeline: `text -> normalizer -> segmenter -> parser -> emotion mapping -> prosody planning -> Piper -> post-processing -> output + evaluation`.

Отже, ми працюємо не з «чистого аркуша», а з уже погодженого **напряму реалізації** і базового монорепо.

## 3. Ключове правило старту
На першій ітерації **не робимо**:
- тренувати власну TTS-модель;
- будувати production-grade мікросервісну інфраструктуру;
- додавати STT як обов'язкову частину;
- оптимізувати все під real-time streaming.

Перший фокус — **керований pipeline text-to-audio**.

## 4. Послідовність, яку було покладено в основу реалізації

### Етап 1. Фіксація контрактів
Було визначено до написання основного коду:
- JSON-формат запиту від front-end до backend;
- JSON-формат відповіді gateway;
- внутрішній контракт metadata між gateway, Python-сервісом і TTS-рівнем;
- формат збереження артефактів у `outputs/`.

### Етап 2. Каркас репозиторію
Каркас у підсумку зібрано у вигляді монорепо з єдиною папкою `src/`:
```text
src/
  apps/
    web/         # React + TypeScript фронтенд
    gateway/     # TypeScript + Fastify gateway backend
  services/
    text-analysis/  # Python FastAPI сервіс аналізу тексту
    tts-adapter/    # Python FastAPI adapter над Piper + ffmpeg
  shared/           # спільні типи/допоміжні модулі для TS

docs/
benchmarks/
reports/
docker/
postman/
```

### Етап 3. Gateway backend
На поточному етапі є мінімальний gateway, який:
- приймає текст через HTTP-ендпоїнт;
- у цільовому варіанті викликає Python-сервіс аналізу;
- викликає TTS-рівень через окремий tts-adapter;
- після синтезу запускає post-processing;
- віддає клієнту URL або шлях до файлу.

### Етап 4. Python-сервіс аналізу
Структура Python-сервісу вже закладена так, щоб перша робоча версія вміла:
- нормалізувати текст;
- сегментувати його;
- знаходити емодзі;
- знаходити `!`, `?`, `...`, повтори символів;
- призначати emotion label та intensity;
- формувати synthesis hints.

### Етап 5. Підключення Piper
TTS-рівень проєкту орієнтовано на Piper. На цьому рівні цільова поведінка така:
- обрати 1 голос для MVP;
- стандартизувати команду синтезу;
- зберігати проміжні WAV-файли;
- зафіксувати поведінку на довгих і коротких сегментах.

### Етап 6. ffmpeg pipeline
Після синтезу сегментів ffmpeg-pipeline у поточній архітектурі має:
- склеїти сегменти;
- вирівняти гучність;
- прибрати зайву тишу, якщо потрібно;
- конвертувати результат у формат для UI.

### Етап 7. Front-end
UI збирається поверх працюючого API. Мінімальний фронтенд у цій архітектурі:
- текстове поле;
- кнопку запуску синтезу;
- блок попереднього перегляду metadata;
- аудіоплеєр;
- повідомлення про помилки.

### Етап 8. Evaluation
Після стабілізації основного потоку передбачено:
- neutral vs expressive comparison;
- збереження метрик;
- базовий benchmark corpus.

## 5. Мінімальний склад MVP
MVP вважається достатнім, якщо є:
- 1 web-форма;
- 1 gateway API;
- 1 Python text-analysis service;
- 1 інтеграція з Piper;
- 1 ffmpeg pipeline;
- 1 каталог із результатами синтезу;
- 1 набір тестових прикладів.

## 7. Технічні пріоритети
Пріоритети, які були покладені в основу каркасу:
1. Контракти між сервісами.
2. Відтворюваний локальний запуск.
3. Працюючий end-to-end pipeline.
4. Якість parsing logic.
5. Якість audio post-processing.
6. UI polish.
7. Benchmarking.

## 8. Чого не робити на старті
- Не змішувати логіку parser з gateway backend.
- Не вбудовувати ffmpeg-команди прямо у front-end.
- Не прив'язувати всю систему лише до одного конкретного API Piper без adapter layer.
- Не зберігати емоцію як неформальні рядки без типізації.
- Не переходити до складного UX, поки немає стабільного API.

## 9. Критерії готовності стартового релізу
Стартовий реліз MVP вважається успішним, якщо:
- локальний запуск відбувається однією командою;
- користувач вводить текст і отримує аудіо;
- metadata формуються перед синтезом;
- нейтральний і виразний режими відрізняються;
- усі артефакти прогнозовано зберігаються у файловій структурі.
